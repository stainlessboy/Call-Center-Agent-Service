Взаимодействие с OpenAI и обработка персональных данных клиентов

Документ для Службы информационной безопасности · AsakaBank · 23 апреля 2026

TL;DR Сервис — Telegram-бот для консультирования клиентов по банковским продуктам (FAQ, подбор кредитов/вкладов/карт, калькулятор платежей, передача оператору). В OpenAI не передаются: ФИО, паспортные данные, ПИНФЛ, ИНН, номера телефонов клиентов, номера карт и счетов, балансы, СМС-коды, пароли. Системный промпт запрещает агенту запрашивать любые персональные данные — для операций, требующих верификации, агент обязан передать клиента живому оператору через тулзу request_operator(reason="identity_required"). Контактные данные лида (имя, телефон), которые клиент вводит после расчёта в калькуляторе, перед попаданием в историю диалога маскируются на токены [NAME] и [PHONE] и в OpenAI уходят уже в обезличенном виде. Сырые имя и телефон сохраняются в локальной PostgreSQL только в таблице leads, инфраструктура полностью размещена в Узбекистане.

1. Что делает сервис

Telegram-бот AsakaBank работает как первая линия консультаций для клиентов. Архитектура: aiogram (Telegram) + FastAPI (вебхук) + LangGraph (оркестрация ИИ-агента) + SQLAlchemy + PostgreSQL (БД). Используется модель OpenAI gpt-4o-mini по умолчанию (настраивается через переменную окружения OPENAI_MODEL).

Бот выполняет следующие задачи:

  • FAQ — отвечает на типовые вопросы о продуктах банка из базы знаний (импортирована из Excel).
  • Подбор продуктов — показывает пользователю ипотеку, автокредит, микрозайм, образовательный кредит, вклады, дебетовые/валютные карты.
  • Калькулятор — детерминированно рассчитывает ежемесячные платежи и доход по вкладу, генерирует PDF-график амортизации.
  • Лид-форма — после расчёта предлагает клиенту перезвон, собирает имя и телефон, сохраняет в таблицу leads.
  • Передача оператору — для операций, требующих верификации личности (блокировка карты, смена пароля, проверка баланса и т. п.) и при явной просьбе клиента.
  • Гибридный режим — оператор может «забрать» сессию и общаться с клиентом напрямую, ИИ при этом не отвечает.

Модель LLM здесь работает как «маршрутизатор и формулятор ответов»: она выбирает, какой инструмент вызвать (например, get_products(category="mortgage") или faq_lookup(query=...)), и формулирует человеческий текст по результату. Все денежные расчёты, поиск по FAQ, сопоставление продуктов и валидация условий выполняются детерминированным кодом, не моделью.

Маскирование применяется в одной точке — функции _finalize_turn() в app/agent/nodes/helpers.py. Она вызывается из узла node_calc_flow с явным параметром mask_user_text="[NAME]" или mask_user_text="[PHONE]" на шагах сбора имени и телефона лида. Это единственное место, где сырая реплика клиента подменяется токеном перед попаданием в историю диалога, откуда она потом ушла бы в OpenAI.

3. Правовая база

  • Закон Республики Узбекистан «О банках и банковской деятельности», ст. 39 — банковской тайной являются сведения об операциях, счетах и вкладах клиентов, позволяющие идентифицировать конкретного клиента. В OpenAI такие сведения не передаются: нет ни идентификатора клиента (ПИНФЛ, паспорт, ФИО), ни номеров счетов/карт, ни сведений о реальных операциях клиента.
  • Закон Республики Узбекистан «О персональных данных» — ЗРУ-547, ст. 27-1 — требование локализации баз данных, содержащих персональные данные граждан Узбекистана, на территории Республики Узбекистан. Все базы данных сервиса (PostgreSQL, чекпоинты LangGraph) размещены в контейнерах внутри инфраструктуры банка в РУз. Сырые персональные данные (имя/телефон лида) хранятся только в этих локальных базах.

Подход сервиса: в OpenAI не передаётся ни одного прямого идентификатора клиента и ни одного персонального данного. Контактные данные лида (имя, телефон) перед отправкой в OpenAI подменяются на обезличенные токены [NAME] и [PHONE]. Системный промпт явно запрещает ИИ-агенту запрашивать у клиента какие-либо персональные данные.

4. Что уходит в OpenAI и что нет

Категория данныхВ OpenAI?Механизм
ФИО клиента НЕТ Бот никогда не запрашивает ФИО — это запрещено системным промптом (см. раздел 5). Если клиент введёт ФИО на шаге лид-формы «Как вас зовут?» — оно маскируется на токен [NAME] перед попаданием в историю диалога (см. раздел 6).
Номер телефона НЕТ Аналогично ФИО: бот не запрашивает телефон в свободном диалоге, а на шаге лид-формы «Укажите ваш номер телефона» введённое значение маскируется на токен [PHONE].
Паспорт, ПИНФЛ, ИНН, дата рождения, адрес НЕТ Бот никогда не запрашивает эти данные. Системный промпт обязывает агента передать клиента оператору через request_operator(reason="identity_required") для любой операции, требующей такой информации.
Номера карт, счетов, IBAN, балансы НЕТ Сервис не имеет интеграций с АБС/процессингом, физически не имеет доступа к балансам и реквизитам клиента. Любая операция со счётом передаётся оператору.
СМС-коды, пароли, ПИН-коды, кодовые слова НЕТ Запрещены системным промптом. Если клиент сам пришлёт такие данные — агент обязан не повторять их в ответе и сразу переключить на оператора.
Telegram user_id, username, first_name НЕТ Telegram-метаданные сохраняются в локальной таблице users для технической идентификации сессий, но не попадают в массив messages, который отправляется в OpenAI (см. app/agent/agent.py).
История сообщений диалога ЧАСТИЧНО Передаются последние 12 сообщений (настраивается через MAX_DIALOG_MESSAGES). Сообщения, в которых клиент мог ввести имя/телефон в lead-форме, уже замаскированы на [NAME]/[PHONE] до попадания в эту историю.
Текст вопросов клиента (FAQ, выбор продукта) ДА Свободный текст пользователя («какие у вас вклады?», «хочу ипотеку на 5 лет») передаётся в OpenAI — это необходимо для понимания запроса. Системный промпт обязывает агента не запрашивать PII в таких вопросах.
Финансовые параметры калькулятора (сумма, срок, первоначальный взнос) ДА Числа передаются в OpenAI для парсинга (отделить «50 млн» от «50 000 000»). Сами по себе числа без идентификатора клиента не образуют банковской тайны и не являются ПД: невозможно сопоставить «10 миллионов на 5 лет» с конкретным человеком, имея только это число.
Названия продуктов банка, ставки, условия ДА Публичная информация о продуктах AsakaBank. Передаётся через результаты тулз get_products(), select_product().
FAQ-ответы из базы знаний ДА Публичная база знаний о продуктах. Передаётся в контекст через тулзу faq_lookup().
Информация о филиалах (адреса, часы) ДА Публичная информация. Передаётся через тулзу find_office().
Системный промпт + dialog state в XML ДА Текст инструкций для агента (роль, политика, правила) и текущее состояние диалога (flow, category, выбранный продукт). Не содержит PII.

5. Защитные меры в системном промпте

Системный промпт ИИ-агента (app/agent/i18n.py) содержит отдельный обязательный раздел, запрещающий запрашивать у клиента любые персональные данные. Раздел продублирован в трёх языковых версиях промпта (русский, английский, узбекский), каждая из которых отправляется в OpenAI в зависимости от языка клиента.

5.1. Русская версия — раздел «ПЕРСОНАЛЬНЫЕ ДАННЫЕ КЛИЕНТА»

## ПЕРСОНАЛЬНЫЕ ДАННЫЕ КЛИЕНТА — СТРОГИЙ ЗАПРЕТ
НИКОГДА не запрашивай у клиента и не предлагай ему прислать: ФИО, паспортные
данные, ПИНФЛ, ИНН, дату рождения, адрес, номер телефона, номер карты или
счёта, баланс, кодовое слово, пароль, ПИН-код, СМС-код, реквизиты документов.

Если для решения вопроса нужны такие данные — НЕ собирай их сам, вызови
`request_operator(reason="identity_required")` и сообщи клиенту, что специалист
поможет с верификацией.

Если клиент сам присылает такие данные — НЕ повторяй их в ответе, не
подтверждай и не запрашивай дополнительно. Сразу переключай на оператора через
`request_operator(reason="identity_required")`.

Контактные данные (имя, телефон) для перезвона запрашиваются ТОЛЬКО встроенной
формой после расчёта в калькуляторе — ты в этом процессе не участвуешь и сам
имя/телефон никогда не спрашиваешь.

5.2. Раздел «ПЕРЕНАПРАВЛЕНИЕ НА ОПЕРАТОРА»

В дополнение, существующий раздел промпта явно перечисляет операции, требующие верификации:

Вызывай `request_operator()` ТОЛЬКО в этих случаях:
1. Клиент ЯВНО просит оператора («позови оператора», «хочу оператора»,
   «живой оператор») — вызывай сразу, без лишних вопросов.
2. Клиент просит операцию, требующую верификации личности (блок/разблок
   карты, смена телефона/пароля, перевод денег, проверка счёта) — вызывай с
   `reason="identity_required"`.
3. Не можешь понять клиента после 2-3 попыток переспросить — вызывай с
   `reason="unclear_message"`.
Эффект для ИБ Любая ситуация, в которой банковская операция требует подтверждения личности клиента, отправляет диалог к живому оператору без сбора PII со стороны ИИ. Промпт обязывает модель НЕ повторять PII даже если клиент прислал его сам — это снижает риск утечки через ответ модели.

6. Маскирование PII в lead-форме

Лид-форма — единственное место в боте, где клиента просят указать имя и телефон. Это происходит после того, как клиент рассчитал параметры кредита/вклада в калькуляторе и согласился на обратный звонок («Да, позвоните мне»).

6.2. Реализация маскирования

Функция _finalize_turn() в app/agent/nodes/helpers.py формирует обновлённый массив сообщений диалога. Она принимает опциональный параметр mask_user_text: если он задан, в качестве HumanMessage добавляется именно это значение, а не сырое сообщение клиента: 

def _finalize_turn(
    state: BotState,
    answer: str,
    dialog: dict,
    keyboard_options: Optional[List[str]] = None,
    *,
    is_fallback: bool = False,
    mask_user_text: Optional[str] = None,
) -> dict:
    user_text = (state.get("last_user_text") or "").strip()
    lang = (dialog or {}).get("last_lang") or "ru"
    msgs = list(state.get("messages") or [SystemMessage(content=get_system_policy(lang))])
    # mask_user_text replaces the raw input before it lands in conversation
    # history that subsequently feeds the LLM (used for lead name/phone steps
    # so PII never leaves the local container in OpenAI requests).
    msgs.append(HumanMessage(content=mask_user_text if mask_user_text is not None else user_text))
    msgs.append(AIMessage(content=answer))

Узел node_calc_flow в app/agent/nodes/calc_flow.py вызывает эту функцию с явным параметром на шагах сбора имени и телефона: 

if lead_step == "name":
    lead_slots = dict(dialog.get("lead_slots") or {})
    lead_slots["name"] = user_text  # сырое имя в локальную память диалога
    new_dialog = {**dialog, "lead_step": "phone", "lead_slots": lead_slots}
    return _finalize_turn(
        state, at("lead_ask_phone", lang), new_dialog,
        mask_user_text="[NAME]"  # ← в OpenAI уйдёт токен, не имя
    )

if lead_step == "phone":
    lead_slots = dict(dialog.get("lead_slots") or {})
    lead_slots["phone"] = user_text  # сырой телефон в локальную память диалога
    try:
        await _save_lead_async({  # сырое значение → в локальную таблицу leads
            ...
            "name": lead_slots.get("name", ""),
            "phone": lead_slots.get("phone", user_text),
        })
        return _finalize_turn(
            state, at("lead_saved", lang), _default_dialog(),
            mask_user_text="[PHONE]"  # ← в OpenAI уйдёт токен
        )

6.3. Что важно понимать про разделение «БД vs OpenAI»

СлойЧто хранитГде находится
PostgreSQL, таблица leads Сырые имя и телефон клиента Локально, контейнер в РУз
PostgreSQL, таблица messages Сырой текст всех сообщений (нужен оператору при подключении и для разбора инцидентов) Локально, контейнер в РУз
LangGraph state в памяти процесса dialog["lead_slots"] с сырыми данными — на время одного запроса Локально, RAM воркера
LangGraph checkpoints (если включён postgres backend) Замаскированные сообщения ([NAME], [PHONE]) + dialog["lead_slots"] с сырыми значениями Локально, та же PostgreSQL
Запрос в OpenAI API Только замаскированные сообщения [NAME], [PHONE] Внешний сервис (OpenAI, США)

Сырые имя и телефон нужны оператору для перезвона клиенту — они физически не могут быть удалены. Маскирование применяется именно перед границей «наша инфраструктура → внешний сервис OpenAI».

7. Передача оператору при идентификации

Если клиент запрашивает операцию, требующую верификации личности (например, «заблокируйте мою карту», «какой у меня баланс», «смените мне пароль»), ИИ-агент обязан вызвать тулзу request_operator с параметром reason="identity_required". Это:

  1. Возвращает клиенту фиксированное локализованное сообщение «Для выполнения этой операции необходима идентификация. Сейчас подключу вас к специалисту».
  2. Помечает сессию в БД флагом show_operator_button = True — на стороне Telegram-бота появляется кнопка «Связаться с оператором».
  3. При нажатии кнопки сессия переходит в human_mode = True — все последующие сообщения клиента сохраняются в БД, но не передаются в OpenAI. Отвечает живой оператор.

Сама тулза получает только короткий тег причины, без свободного текста:

@lc_tool
async def request_operator(
    reason: str = "",
    state: Annotated[dict, InjectedState] = None,
) -> str:
    """Transfer the user to a live operator.
    reason: short tag — "identity_required" / "unclear_message" / "user_request".
    """
    ...
    if "identity" in reason.lower():
        return at("operator_identity_required", lang)
Эффект для ИБ Параметр reason — это короткий технический тег (identity_required, unclear_message, user_request), а не свободный текст с описанием запроса. Соответственно, описание операции клиента не уходит в OpenAI повторно через тулзу.

8. Живой пример: как лид попадает (или не попадает) в OpenAI

8.1. Сценарий

Клиент рассчитывает ипотеку, соглашается на перезвон, вводит имя и телефон, после чего задаёт ещё один вопрос.

8.2. Сообщения, которые накапливаются в state["messages"]

До маскирования (как было бы без правки):

SystemMessage:  "## РОЛЬ\nТы опытный консультант банка..."

Ключевой принцип Сырые ПД клиента доступны менеджеру AsakaBank для перезвона (для этого их и собирали) и хранятся только в инфраструктуре банка в Узбекистане. Внешний сервис OpenAI не получает ни имени, ни телефона клиента — только обезличенные токены.

9. Что хранится локально в Узбекистане

КомпонентГде размещёнЧто хранит
PostgreSQL 16 Docker-контейнер в РУз (см. docker-compose.yml) Все данные сервиса: users, chat_sessions, messages (сырой текст), leads (сырые имя/телефон), credit_product_offers, deposit_product_offers, card_product_offers, faq_items, branches, чекпоинты LangGraph
FastAPI + LangGraph (Python) Docker-контейнер в РУз Бизнес-логика, обработка вебхуков, оркестрация агента. Состояние диалога — в памяти процесса и в чекпоинтах PostgreSQL.
SQLAdmin панель (/admin) Тот же контейнер FastAPI Веб-интерфейс для операторов и администраторов. Аутентификация через ENV (ADMIN_USERNAME/ADMIN_PASSWORD).
nginx (опционально) Контейнер в РУз Reverse proxy, TLS termination для вебхука Telegram.

Внешние зависимости сервиса:

  • Telegram Bot API — для приёма сообщений клиентов и отправки ответов. Это требование платформы; переписка с клиентом физически проходит через серверы Telegram.
  • OpenAI API — единственный внешний ИИ-сервис. Передаются обезличенные данные согласно разделу 4.
  • ЦБ Узбекистана — публичный API курсов валют (cbu.uz), без передачи каких-либо данных клиента.

Важно про чекпоинты LangGraph При LANGGRAPH_CHECKPOINT_BACKEND=postgres в чекпоинты сохраняется массив messages, где имя/телефон лида уже замаскированы ([NAME]/[PHONE]). Сырые значения сохраняются только в таблице leads и в dialog["lead_slots"] (которое тоже попадает в чекпоинт, но физически находится только в локальной БД).
